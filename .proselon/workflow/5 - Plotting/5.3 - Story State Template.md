# Story State Ledger Template

One rolling ledger per book: `Plot/Book N/Story State.md`. It is the authoritative record of *in-story state* — where everyone is, what they know, what's broken, what's been promised to the reader — as of the most recently completed scene. It also carries one piece of workflow state: the per-scene Production table (see Production Tracking below).

The ledger exists to replace chained state. Without it, scene-to-scene continuity passes through each previous scene plot like a linked list, and a condition set in chapter 2 must be hand-copied through dozens of scene plots — or silently dropped. With the ledger, any scene's state context is one file read.

## Lifecycle

- **Create** the ledger when the book plot is created (Step 5). Most sections start empty or carry initial conditions from the book plot.
- **For Book 2 onward**, seed from the previous book's final `Story State.md`. This is a once-per-book, high-blast-radius operation — work it as a checklist, not from memory:
  1. Read the previous book's final ledger end to end.
  2. Carry forward every open (unpaid, undropped) thread with its original `B#C#S#` provenance, and every ongoing condition that survives the gap between books.
  3. Seed each character's entry from their end-state in the previous ledger — location, physical, possessions, knowledge, emotional through-line — adjusted for time passed between books.
  4. Migrate any stable facts still in the old ledger's Established Facts Index into character profiles or `Worldbuilding/` (see Update Discipline), then mark the previous book's ledger closed with a pointer line to the new one.
- **Update** it as the final act of completing every scene — after the editing passes, before reporting. Record the scene's deltas: time passed, injuries, locations, possessions, knowledge changes, new established facts, threads planted or paid — and advance the `Last updated after:` line. Mark the scene's Ledger cell in the Production table at the same time.
- **Load** it for every scene plot (Step 5), every continuity check (6.5), and any revision or continuity task. The passes load it only when their pass doc says to.

## Out-of-Order Work

The ledger assumes scenes complete in reading order — its state sections describe the world "as of the most recently completed scene," a model that cannot represent two frontiers at once. If the author directs out-of-order drafting ("skip to Chapter 12"):

- Tell the author continuity checking degrades until the gap closes — that's the trade, not a blocker.
- Keep the ledger's state sections and `Last updated after:` pinned to the furthest **contiguous** completed scene.
- Record the out-of-order scenes' deltas in a quarantined `## Ahead of Ledger` section (same entry format, keyed by scene ID). When the gap closes, fold those deltas into the main sections in reading order and delete the quarantine section.
- Scene plots and continuity checks for out-of-order scenes must read both the main sections and the quarantine section.

## Update Discipline

- Update deltas only — do not rewrite the whole file each scene.
- Keep entries to one line each. The ledger is a lookup table, not prose.
- When a condition resolves (an injury heals, a thread pays off), update its entry — mark threads `paid` rather than deleting them, so promise/payoff remains auditable. A thread the author deliberately abandons is marked `dropped` with a one-word reason — never left `open` forever.
- Cite scene IDs (`B#C#S#`) so any entry can be traced to its source.
- If Open Threads grows past comfortable scanning, split it to `Plot/Book N/Threads.md` using the same entry format and leave a pointer in the ledger.
- When a fact in the Established Facts Index has become stable canon — rendered, unchallenged, and several chapters old — migrate it at act boundaries into the relevant character profile or `Worldbuilding/` file (keep its `B#C#S#` provenance), where it loads only when relevant. Facts with no natural home roll up to `Plot/Book N/Facts Archive.md`, with a pointer line left in the ledger. The Index holds only recent and at-risk facts, so it stays scannable.

## Production Tracking

The `Production` section is the per-scene workflow record — which passes have run. It is maintained by the orchestrating agent; editing-pass subagents ignore it (it is workflow state, not in-story state).

- Add a row when a scene plot is created (mark Plot). Mark Draft when the manuscript file saves, each pass cell (Dev, Voice, Line, Cont, Copy, Proof) when that pass completes, and Ledger when the scene's ledger update is done.
- **Marking vocabulary:** completed cells are `x`; pending cells are `—`. Use no other symbols — resume logic depends on unambiguous marks.
- A scene is finished only when its entire row is marked.
- Resume interrupted work at the first unmarked cell. Never re-run a marked pass — the Voice Rewrite (6.3) especially is a generative rewrite, and re-running it on already-voiced prose compounds drift.
- If the `Last updated after:` line and the Production table disagree, **the Production table is authoritative** — it is marked per pass; the pointer line is a convenience. Correct the pointer to match.

## Completion Criteria

A healthy ledger:

- Has a `Last updated after:` line matching the most recently completed manuscript scene.
- Covers every character who has appeared on the page in this book.
- Records first-rendering scene IDs for established facts, so the Information Log in scene plots can cite provenance without re-reading prior scene plots.
- Tracks every open promise to the reader with a planned payoff location.
- Has a Production row for every scene plot that exists, accurate to the last completed pass.
- Contains no prose, no commentary, no duplicated canon (character backstory lives in profiles; the ledger holds only *current state*).

**Output saves to:** `Plot/Book N/Story State.md`
